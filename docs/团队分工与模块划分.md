# MoeNews 团队分工与模块划分说明

本说明用于明确 MoeNews 系统在开发过程中的团队分工与模块划分。系统采用前后端分离架构，后端基于 Spring Boot 与 MyBatis-Plus，前端基于 Vue 3。开发团队共 3 人，采用按技术架构层级横向切分的协作模式。

---

## 一、 团队角色定位与职责概要

### 成员 A：前端交互与控制层开发（前端与 Controller）
*   **职责定位**：负责系统的表现层与接口接入层，是前后端沟通的桥梁。
*   **核心工作**：
    *   搭建前端工程脚手架，配置路由守卫与状态管理。
    *   实现前台新闻展示、后台管理系统的全部用户界面与交互微动画。
    *   在后端定义所有控制层接口，进行入参合法性校验，并使用统一返回结果对象封装数据。

### 成员 B：业务逻辑与安全控制开发（Service与Security）
*   **职责定位**：负责系统的核心业务规则与系统安全性。
*   **核心工作**：
    *   设计并实现业务逻辑层接口及对应的实现类。
    *   实现基于 JWT 的用户身份校验与无状态认证。
    *   编写安全拦截器，处理越权和未登录拦截。
    *   定义数据传输对象与视图对象，管理全局自定义异常与统一异常处理。

### 成员 C：数据持久化与底层支撑开发（Mapper与Database与Config）
*   **职责定位**：负责系统的基础架构、数据安全及部署支撑。
*   **核心工作**：
    *   负责数据库物理模型设计，编写表结构初始化与演示数据脚本。
    *   配置 MyBatis-Plus，实现数据访问层的 CRUD 接口。
    *   定义数据库实体类及公共基类，处理公共字段自动填充（如创建时间、更新时间）。
    *   编写本地文件存储服务，提供图片上传支撑，并编写项目构建与全局配置文件。

## 三、 功能开发协作流转流程

以系统中核心的 **发布新闻** 功能为例，展示三人之间的协作流转机制：

```
[步骤 1: 成员 C] -> 设计 news 表 -> 创建 News 实体类 -> 编写 NewsMapper 接口
                       │
                       ▼
[步骤 2: 成员 B] -> 定义 NewsRequest (DTO) -> 编写 NewsService 业务逻辑及校验
                       │
                       ▼
[步骤 3: 成员 A] -> 编写 AdminNewsController 接收请求 -> 编写前端编辑页面与 API 对接
```

1.  **数据层支撑（成员 C）**：
    在数据库中增加 `news` 表，并在 `com/moenews/entity/News.java` 中定义好字段，随后在 `com/moenews/mapper/NewsMapper.java` 中建立持久层接口，支持数据的插入。
2.  **业务层处理（成员 B）**：
    在 `com/moenews/dto/NewsRequest.java` 中定义前端提交的数据格式（标题、内容、分类等），并在 `com/moenews/service/impl/NewsServiceImpl.java` 中实现具体的发布逻辑（包括分类合法性验证、敏感字处理、调用持久层入库等）。
3.  **表现层呈现（成员 A）**：
    在 `com/moenews/controller/AdminNewsController.java` 中开放 `/api/admin/news` 控制器接口，接收前端参数并调用业务层。同时在前端 `frontend/src/views/admin/NewsEditView.vue` 页面中完成 Markdown 编辑器的接入和数据的异步提交，完成联调。
