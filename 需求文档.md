# 新闻发布与管理系统需求规格说明书

## 1. 引言

### 1.1 项目名称

新闻发布与管理系统

### 1.2 项目背景

随着互联网信息传播的发展，新闻内容的发布、分类、维护和评论管理成为信息管理系统中的常见需求。传统人工管理新闻内容的方式效率较低，且不便于对新闻状态、用户评论、新闻分类和后台用户进行统一管理。因此，本项目设计并实现一个基于 Java 企业级开发技术的新闻发布与管理系统，用于完成新闻内容发布、新闻分类维护、用户管理、评论管理和前台新闻展示等功能。

本系统采用前后端分离架构，后端使用 Spring Boot 提供 RESTful API，前端使用 Vue 3 和 Naive UI 构建页面，数据库使用 MySQL 存储业务数据。系统既包含普通用户端，也包含管理员后台，能够体现 Java 企业级 Web 应用开发、数据库应用、MVC 分层设计和系统工程实现能力。

### 1.3 编写目的

本文档用于明确新闻发布与管理系统的功能需求、非功能需求、用户角色、业务规则、数据需求、系统边界和验收标准，为后续系统设计、数据库设计、接口设计、编码实现、测试和课程设计报告撰写提供依据。

### 1.4 项目目标

本项目的主要目标如下：

1. 实现一个功能完整、结构清晰的新闻发布与管理系统。
2. 支持普通用户注册、登录、浏览新闻、查看新闻详情和发表评论。
3. 支持管理员对用户、新闻分类、新闻内容、评论和系统统计信息进行管理。
4. 支持新闻 Markdown 编辑、封面上传、分页查询、模糊搜索和状态管理。
5. 采用 Spring Boot、MyBatis-Plus、MySQL、Vue 3、Naive UI 等技术完成前后端分离系统开发。
6. 通过 JWT、自定义拦截器、BCrypt、参数校验等方式实现基本安全控制。
7. 通过分层架构、统一返回结果、统一异常处理、DTO/VO 分离等方式提高系统可维护性和可扩展性。
8. 前端采用轻量自定义布局，不直接使用大型后台模板，以降低前端工程复杂度，提高项目可控性。

## 2. 系统概述

### 2.1 系统定位

本系统定位为一个新闻发布与管理系统，包含前台新闻展示和后台内容管理两个部分。

前台主要面向普通用户，提供注册、登录、新闻浏览、新闻详情查看和评论发布功能。

后台主要面向管理员，提供用户管理、新闻分类管理、新闻管理、评论管理、封面上传和首页统计功能。

系统不定位为大型新闻门户平台，不实现新闻推荐算法、复杂审核流、全文搜索引擎、消息通知、点赞收藏、复杂 RBAC 权限系统等高级功能。

### 2.2 系统架构

系统采用前后端分离架构。

后端负责业务逻辑处理、数据访问、安全认证、文件上传和接口响应。

前端负责页面展示、表单交互、数据请求、路由控制和用户体验优化。

系统整体结构如下：

```text
前端 Vue 3 + Naive UI
  ↓ Axios 请求
后端 Spring Boot RESTful API
  ↓ MyBatis-Plus
MySQL 数据库
```

### 2.3 技术栈

#### 2.3.1 后端技术栈

| 技术                | 作用                  |
| ----------------- | ------------------- |
| Spring Boot       | 后端主框架               |
| Spring Web        | 提供 RESTful API      |
| MyBatis-Plus      | 数据库 ORM、Mapper、分页查询 |

| 自定义拦截器            | 登录校验和角色权限控制         |

说明：本系统不引入完整 Spring Security 权限框架，仅使用 JWT、自定义拦截器和 BCrypt 实现轻量级认证授权方案。

#### 2.3.2 前端技术栈

| 技术           | 作用                          |

## 3. 用户角色分析

### 3.1 普通用户 USER

普通用户可以通过系统前台完成注册和登录，登录后可以浏览新闻、查看新闻详情、查看评论、发表评论和删除自己的评论。

普通用户不能进入后台管理页面，不能访问后台管理接口，不能发布新闻，不能管理新闻分类、用户和其他用户评论。

### 3.2 管理员 ADMIN

管理员拥有系统后台管理权限，可以管理用户、新闻分类、新闻内容和评论，也可以查看首页统计数据。

管理员同时具备新闻编辑员能力，可以直接编辑、发布、下架和删除新闻。因此本系统不单独设计编辑员角色，编辑员权限合并到管理员角色中。

### 3.3 编辑员角色说明

本系统当前阶段不单独实现 EDITOR 编辑员角色。新闻编辑、新闻发布、新闻下架等操作统一由管理员完成。

系统在用户表中保留 role 字段，当前取值为 ADMIN 和 USER。后续如果系统规模扩大，可以在现有 role 字段基础上扩展 EDITOR 角色。

## 5. 详细功能需求

### 5.1 注册登录模块

#### 5.1.1 普通用户注册

普通用户可以在前台注册账号。

注册时需要填写用户名、密码、确认密码和昵称。

系统应完成以下校验：

1. 用户名不能为空。
2. 密码不能为空。
3. 两次输入的密码必须一致。
4. 用户名不能重复。
5. 注册用户默认角色为 USER。
6. 注册用户默认状态为启用。

注册成功后，用户可以跳转到登录页面进行登录。

#### 5.1.2 用户登录

普通用户和管理员共用登录入口。

用户输入用户名和密码后，系统进行身份校验。

登录规则如下：

1. 用户名和密码不能为空。
2. 用户名不存在时提示登录失败。
3. 密码错误时提示登录失败。
4. 账号被禁用时不允许登录。
5. 登录成功后，后端生成 JWT Token 并返回用户基本信息。
6. 前端保存 Token，并在后续请求中通过请求头携带 Token。

登录成功后的跳转规则如下：

| 角色    | 登录后跳转   |
| ----- | ------- |
| ADMIN | 后台首页    |
| USER  | 前台新闻列表页 |

#### 5.1.3 退出登录

用户点击退出登录后，前端清除本地保存的 Token 和用户信息，并跳转到登录页面。

#### 5.1.4 获取当前用户信息

登录用户可以获取当前登录用户的基本信息，包括用户 ID、用户名、昵称、角色、状态等。

系统接口不应返回用户密码字段。

### 5.2 前台新闻模块

#### 5.2.1 新闻列表

游客和普通用户均可访问新闻列表页面。

新闻列表应支持：

1. 分页查询。
2. 标题关键词模糊搜索。
3. 新闻分类筛选。
4. 按发布时间倒序展示。
5. 仅展示已发布状态的新闻。
6. 列表页展示新闻标题、摘要、封面、分类、发布时间、浏览量等信息。
7. 列表页不返回完整新闻正文，以减少数据传输量。

#### 5.2.2 新闻详情

游客和普通用户均可访问新闻详情页面。

新闻详情页应展示：

1. 新闻标题。
2. 新闻分类。
3. 新闻封面。
4. 发布时间。
5. 浏览量。
6. Markdown 渲染后的新闻正文。
7. 新闻评论列表。

用户访问新闻详情时，系统可以增加新闻浏览量。

#### 5.2.3 Markdown 内容展示

新闻正文采用 Markdown 格式存储。

前台新闻详情页面使用 Markdown 预览组件将 Markdown 原文渲染为可阅读内容。

系统只允许管理员发布新闻正文，普通用户评论不支持 Markdown，仅支持纯文本，以降低安全风险和实现复杂度。

---

### 5.3 评论模块

#### 5.3.1 查看评论

游客和普通用户均可以查看新闻详情页下的评论列表。

评论列表展示内容包括：

1. 评论用户昵称。
2. 评论内容。
3. 评论时间。
4. 评论状态为显示的评论。

#### 5.3.2 发表评论

普通用户登录后可以在新闻详情页发表评论。

发表评论规则如下：

1. 用户必须登录。
2. 评论必须绑定有效新闻。
3. 评论内容不能为空。
4. 评论内容长度限制为 500 字以内。
5. 评论默认为显示状态。
6. 评论不支持 Markdown，仅支持纯文本。

#### 5.3.3 删除自己的评论

普通用户可以删除自己发布的评论。

删除规则如下：

1. 用户必须登录。
2. 用户只能删除自己的评论。
3. 普通用户不能删除其他用户的评论。
4. 删除可采用逻辑删除方式。

#### 5.3.4 管理员评论管理

管理员可以在后台管理所有评论。

管理员评论管理功能包括：

1. 分页查询评论。
2. 按评论内容模糊搜索。
3. 按评论状态筛选。
4. 删除评论。
5. 隐藏评论。
6. 显示评论。

评论状态设计如下：

| 状态值 | 状态说明 |
| --- | ---- |
| 0   | 隐藏   |
| 1   | 显示   |

### 5.4 用户管理模块

用户管理模块由管理员使用。

#### 5.4.1 用户列表

管理员可以分页查询用户列表。

用户列表支持：

1. 按用户名模糊搜索。
2. 按角色筛选。
3. 按状态筛选。
4. 分页展示。

#### 5.4.2 新增用户

管理员可以新增后台用户或普通用户。

新增用户时应校验：

1. 用户名不能为空。
2. 密码不能为空。
3. 用户名不能重复。
4. 角色必须合法。
5. 状态必须合法。

#### 5.4.3 修改用户

管理员可以修改用户的昵称、邮箱、手机号、角色和状态等信息。

用户名原则上不建议频繁修改。

系统不允许直接修改用户密码字段，密码修改应通过重置密码功能完成。

#### 5.4.4 删除用户

管理员可以删除用户。

删除规则如下：

1. 管理员不能删除当前登录账号。
2. 删除用户时应考虑其发布的新闻和评论。
3. 可采用逻辑删除方式，避免数据关系丢失。

#### 5.4.5 启用和禁用用户

管理员可以启用或禁用用户。

用户状态设计如下：

| 状态值 | 状态说明 |
| --- | ---- |
| 0   | 禁用   |
| 1   | 启用   |

禁用用户不能登录系统。

#### 5.4.6 重置密码

管理员可以重置用户密码。

重置后的密码应使用 BCrypt 加密后再保存到数据库。

#### 5.4.7 普通用户个人信息维护

普通用户登录后可以查看和修改自己的基本信息，如昵称、邮箱、手机号等。

普通用户不能修改自己的角色和状态。

### 5.5 新闻分类管理模块

新闻分类管理模块由管理员使用。

#### 5.5.1 分类列表

管理员可以查看新闻分类列表。

分类列表支持：

1. 按分类名称模糊搜索。
2. 按状态筛选。
3. 按排序字段展示。
4. 分页查询或普通列表查询。

#### 5.5.2 新增分类

管理员可以新增新闻分类。

新增分类时应校验：

1. 分类名称不能为空。
2. 分类名称不能重复。
3. 分类状态必须合法。
4. 排序值应为有效数字。

#### 5.5.3 修改分类

管理员可以修改分类名称、分类描述、排序值和状态。

#### 5.5.4 删除分类

管理员可以删除分类。

删除规则如下：

1. 如果分类下存在新闻，不允许直接删除该分类。
2. 删除前应进行确认提示。
3. 分类删除可采用逻辑删除方式。

#### 5.5.5 启用和禁用分类

管理员可以启用或禁用分类。

分类状态设计如下：

| 状态值 | 状态说明 |
| --- | ---- |
| 0   | 禁用   |
| 1   | 启用   |

禁用分类不能用于新发布新闻。

### 5.6 新闻管理模块

新闻管理模块由管理员使用。

#### 5.6.1 新闻列表

管理员可以分页查询新闻列表。

新闻列表支持：

1. 按标题模糊搜索。
2. 按分类筛选。
3. 按状态筛选。
4. 按发布时间倒序展示。
5. 分页查询。

新闻列表展示字段包括：

1. 新闻 ID。
2. 新闻标题。
3. 新闻摘要。
4. 新闻分类。
5. 新闻封面。
6. 新闻状态。
7. 浏览量。
8. 创建时间。
9. 更新时间。

#### 5.6.2 发布新闻

管理员可以发布新闻。

发布新闻需要填写：

1. 新闻标题。
2. 新闻摘要。
3. 新闻正文。
4. 新闻分类。
5. 新闻封面。
6. 新闻状态。

发布新闻时应校验：

1. 标题不能为空。
2. 正文不能为空。
3. 分类不能为空。
4. 分类必须存在且处于启用状态。
5. 新闻状态必须合法。

#### 5.6.3 Markdown 编辑新闻正文

新闻正文使用 Markdown 格式编辑和保存。

后台新闻编辑页使用 md-editor-v3 作为 Markdown 编辑器，支持 Markdown 输入和实时预览。

数据库保存 Markdown 原文，不保存转换后的 HTML。

这样可以保证正文内容便于二次编辑，也可以降低后端处理复杂度。

#### 5.6.4 修改新闻

管理员可以修改新闻标题、摘要、正文、分类、封面和状态等信息。

#### 5.6.5 删除新闻

管理员可以删除新闻。

新闻删除采用逻辑删除方式。

新闻删除后，前台不再展示该新闻。该新闻关联评论可保留但不在前台展示，也可以在后台按需处理。

#### 5.6.6 新闻状态管理

新闻状态设计如下：

| 状态值 | 状态说明 |
| --- | ---- |
| 0   | 草稿   |
| 1   | 已发布  |
| 2   | 已下架  |

前台新闻列表和详情页只展示已发布新闻。

管理员可以将新闻设置为草稿、已发布或已下架。

#### 5.6.7 新闻浏览量

系统记录新闻浏览量。

用户访问新闻详情时，系统可以对新闻浏览量进行增加。

---

### 5.7 文件上传模块

#### 5.7.1 新闻封面上传

管理员可以上传新闻封面图片。

上传规则如下：

1. 只允许上传 jpg、jpeg、png 格式图片。
2. 文件大小限制为 2MB 以内。
3. 文件保存到服务器本地 uploads 目录。
4. 数据库只保存图片访问路径，不保存图片二进制内容。
5. 上传成功后，系统返回图片访问地址。

#### 5.7.2 文件存储扩展说明

当前系统采用本地文件存储方式，便于课程设计开发和部署。

系统可以预留 FileStorageService 接口，后续可扩展为七牛云、阿里云 OSS 等对象存储方式。

### 5.8 首页统计模块

首页统计模块由管理员使用。

后台首页展示系统关键数据统计，包括：

1. 用户总数。
2. 新闻总数。
3. 分类总数。
4. 评论总数。
5. 已发布新闻数。
6. 草稿新闻数。
7. 已下架新闻数。

首页统计数据用于帮助管理员快速了解系统运行情况。

## 6. 非功能需求

### 6.1 安全性需求

系统应具备基本安全控制能力。

具体要求如下：

1. 用户密码不能明文存储，应使用 BCrypt 加密。
2. 用户登录成功后使用 JWT Token 表示登录状态。
3. 前端请求需要登录的接口时，应在请求头中携带 Token。
4. 后端通过自定义拦截器校验 Token 是否有效。
5. 后台管理接口仅允许 ADMIN 角色访问。
6. 普通用户不能访问后台管理接口。
7. 评论发布、删除自己的评论、个人信息修改等操作必须登录后才能执行。
8. 文件上传必须限制文件格式和大小。
9. 后端使用 Spring Validation 对请求参数进行校验。
10. 接口返回用户信息时不允许返回密码字段。

### 6.2 性能需求

系统应保证常用操作具有较好的响应速度。

具体要求如下：

1. 用户、新闻、评论等列表采用分页查询，避免一次性加载大量数据。
2. 新闻列表页只返回摘要信息，不返回完整正文。
3. 新闻详情页再返回完整 Markdown 正文。
4. 新闻列表默认按照发布时间倒序展示。
5. 常用查询字段应建立索引，如 username、category_id、status、create_time。
6. 首页统计只查询必要的数量信息。
7. 图片内容不直接存入数据库，数据库仅保存图片路径。

### 6.3 易用性需求

系统界面应清晰、直观、易操作。

具体要求如下：

1. 前台和后台页面结构清晰。
2. 后台采用左侧菜单、顶部栏和内容区域布局。
3. 前端页面采用 Naive UI 组件构建，保证界面风格统一、视觉清爽。
4. 表格数据支持搜索、筛选和分页。
5. 新增、修改、删除操作后应给出成功或失败提示。
6. 删除操作前应弹出确认提示。
7. 表单输入错误时应给出明确提示。
8. 新闻状态、用户状态、评论状态应使用明显标签展示。
9. Markdown 编辑器应支持编辑和预览，方便管理员编写新闻正文。

### 6.4 可维护性需求

系统代码应具有良好的分层结构和统一规范。

具体要求如下：

1. 后端采用 Controller、Service、Mapper 分层结构。
2. Controller 层负责接收请求和返回响应。
3. Service 层负责业务逻辑处理。
4. Mapper 层负责数据库访问。
5. 使用 Entity 表示数据库实体。
6. 使用 DTO 接收前端请求参数。
7. 使用 VO 返回前端展示数据。
8. 使用 Result 统一接口返回格式。
9. 使用 GlobalExceptionHandler 统一异常处理。
10. 使用枚举类管理角色、状态等固定值。
11. 使用 BaseEntity 统一 id、createTime、updateTime、deleted 等公共字段。
12. 前端按照 api、views、router、stores、components、utils、layout 等目录进行模块化组织。
13. 前端不直接使用大型后台模板，避免模板结构过重导致维护困难。

### 6.5 可扩展性需求

系统设计应保留后续扩展能力。

可扩展方向包括：

1. 当前角色为 ADMIN 和 USER，后续可扩展 EDITOR 编辑员角色。
2. 当前评论为一级评论，后续可扩展评论回复、点赞和举报。
3. 当前新闻状态为草稿、已发布、已下架，后续可扩展审核中、审核驳回等状态。
4. 当前文件上传采用本地存储，后续可扩展对象存储服务。
5. 当前搜索采用数据库 like 查询，后续可扩展全文搜索。
6. 当前权限采用角色字段判断，后续可扩展角色权限表和菜单权限。
7. 当前前台为基础新闻展示，后续可扩展新闻推荐、热门新闻、收藏等功能。
8. 当前前端采用轻量自定义布局，后续可以根据需要接入更完整的后台管理模板。

### 6.6 数据完整性需求

系统应保证数据关系正确、字段合法、状态有效。

具体要求如下：

1. 用户名必须唯一。
2. 分类名称必须唯一。
3. 新闻必须绑定有效分类。
4. 新闻必须绑定有效作者。
5. 评论必须绑定有效新闻。
6. 评论必须绑定有效用户。
7. 禁用分类不能用于发布新闻。
8. 禁用用户不能登录系统。
9. 删除分类前应判断分类下是否存在新闻。
10. 删除用户时应考虑其新闻和评论数据。
11. 状态字段必须使用合法枚举值。
12. 数据库使用逻辑删除字段，避免误删导致数据丢失。

## 12. 业务规则

### 12.1 用户业务规则

1. 用户名必须唯一。
2. 用户密码必须加密保存。
3. 禁用用户不能登录。
4. 管理员不能删除当前登录账号。
5. 普通用户不能修改自己的角色和状态。
6. 管理员可以重置用户密码。

### 12.2 新闻分类业务规则

1. 分类名称不能为空。
2. 分类名称不能重复。
3. 禁用分类不能用于发布新闻。
4. 分类下存在新闻时，不允许直接删除分类。
5. 分类按照 sort 字段排序。

### 12.3 新闻业务规则

1. 新闻标题不能为空。
2. 新闻正文不能为空。
3. 新闻必须选择有效分类。
4. 新闻正文以 Markdown 格式保存。
5. 前台只展示已发布新闻。
6. 新闻删除采用逻辑删除。
7. 新闻列表不返回完整正文。
8. 新闻详情展示完整 Markdown 渲染内容。

### 12.4 评论业务规则

1. 只有登录用户才能发表评论。
2. 评论内容不能为空。
3. 评论内容长度不能超过 500 字。
4. 评论必须绑定有效新闻。
5. 用户只能删除自己的评论。
6. 管理员可以删除、隐藏或显示所有评论。
7. 评论只支持纯文本，不支持 Markdown。

### 12.5 文件上传业务规则

1. 只允许上传 jpg、jpeg、png 图片。
2. 上传文件大小限制为 2MB。
3. 图片保存到本地 uploads 目录。
4. 数据库只保存图片路径。
5. 上传失败时应返回明确错误信息。

## 15. 项目开发优先级

系统建议按以下顺序开发：

1. 数据库建表。
2. 后端项目初始化。
3. 统一返回结果 Result。
4. 全局异常处理。
5. MyBatis-Plus 配置。
6. 注册、登录、JWT 生成与解析。
7. 自定义拦截器和权限控制。
8. 新闻分类管理。
9. 新闻管理。
10. Markdown 编辑器接入。
11. 前台新闻列表和新闻详情。
12. 评论功能。
13. 用户管理。
14. 文件上传。
15. 首页统计。
16. 前端页面完善。
17. 系统测试。
18. 课程设计报告整理。

前端开发建议优先完成基础布局、登录注册页面、新闻列表页、新闻详情页和后台通用表格页面，再逐步完善新闻编辑、评论管理和统计首页。

## 16. 总结

本系统最终确定为一个基于 Spring Boot 和 Vue 3 的新闻发布与管理系统。系统包括普通用户端和管理员后台，普通用户可以注册、登录、浏览新闻、查看新闻详情和发表评论，管理员可以管理用户、新闻分类、新闻内容、评论和系统统计数据。

系统后端采用 Spring Boot、Spring Web、MyBatis-Plus、MySQL、JWT、自定义拦截器、BCrypt 和 Spring Validation 实现业务功能、安全认证、权限控制和参数校验。系统前端采用 Vue 3、Vite、Naive UI、Axios、Vue Router、Pinia 和 md-editor-v3 实现前台展示、后台管理和 Markdown 新闻编辑功能。

本系统不引入完整 Spring Security，不直接采用大型后台模板，而是采用轻量、清晰、可控的工程方案。通过统一返回结果、统一异常处理、分层架构、DTO/VO 分离、分页查询、逻辑删除、状态管理、数据完整性校验和轻量自定义前端布局，系统具有较好的安全性、易用性、可维护性和可扩展性，适合 Java 企业级应用开发课程设计的实现与答辩展示。
